## 明确国外路线

国外路线包括：

- 海外静态托管、SSR/Serverless 平台和应用托管平台；
- 中国香港、澳门、台湾及其他境外地区的 VPS 或云服务器；
- 主要服务全球用户或使用国际域名的项目。

“国外上线”描述托管入口，不自动代表用户、公司主体或数据都在国外。先确认：

1. 主要用户在哪些国家或地区；
2. 是否仍需要服务中国大陆用户；
3. 应用、数据库、对象存储和第三方 API 的区域；
4. 是否涉及登录、支付、用户内容、敏感数据或受监管行业。

## 选择托管平台或 VPS

| 项目特征 | 优先考虑 | 原因 |
|---|---|---|
| 纯静态站、文档、营销页 | 静态托管/CDN 平台 | 构建、全球分发和证书通常由平台管理 |
| SPA | 静态托管平台 | 重点处理构建目录和 SPA 路由回退 |
| 常见 SSR 框架 | 明确支持该框架的托管平台 | 减少运行时和边缘入口配置 |
| 短时 API/函数 | Serverless 或应用平台 | 按平台函数模型部署 |
| 常驻 API、WebSocket、后台任务 | 支持常驻服务的应用平台或 VPS | 避免函数时长和生命周期限制 |
| 自定义系统包、复杂网络、完整主机控制 | VPS/云主机 | 需要操作系统级权限 |
| 本地磁盘上传或状态 | 对象存储/托管数据库，必要时 VPS | 多实例与重新部署不能依赖临时磁盘 |

平台示例只能用于建立类别概念，不作为默认推荐。用户点名 Vercel、Cloudflare、Netlify、Render、Railway、Fly.io、AWS、Azure、Google Cloud、DigitalOcean 等服务时，先查看该服务当前官方文档、区域、运行时和限制。

优先选择最少运维且能满足真实需求的方案。不要因为熟悉 Nginx 就给纯静态站购买 VPS，也不要把必须常驻的服务硬塞进不相容的函数平台。

## 托管平台完整流程

### 1. 建立平台部署卡

记录：

- 平台、账号和项目/团队
- GitHub 仓库、生产分支和提交 SHA
- 根目录、包管理器、运行时版本
- 安装、构建、启动命令和产物目录
- 环境变量名称及预览/生产作用域
- 数据库、对象存储、队列、后台任务
- 目标区域和平台临时域名

### 2. 连接 GitHub

使用平台官方 GitHub App/OAuth 集成，只授权目标仓库。确认生产分支和自动部署策略。不要向第三方页面粘贴个人访问令牌，除非官方文档明确要求且已限制最小权限。

### 3. 配置构建

让平台配置与本地生产构建保持一致：

- 使用仓库锁文件；
- 固定运行时主版本；
- 正确选择 monorepo 根目录；
- 明确构建命令和输出目录；
- 区分构建时变量与运行时变量；
- 不把服务端秘密暴露为前端公共变量。

先部署一个预览版本。构建失败时先读完整日志，不要连续修改多个字段。

### 4. 配置运行与数据

- 把应用、数据库和对象存储放在合理相近的区域；
- 明确文件系统是否临时；
- 明确函数超时、内存、并发、休眠和冷启动行为；
- 确认 WebSocket、定时任务、队列和后台 Worker 是否受支持；
- 为数据库迁移建立一次性、可审计的执行步骤；
- 不在多个并发实例启动时自动执行破坏性迁移。

### 5. 验证预览部署

通过平台临时域名验证：

1. 首页和静态资源；
2. SPA/SSR 深链接；
3. 健康检查与一条真实 API；
4. 登录、回调、上传、下载和第三方 API；
5. 日志中没有秘密和持续错误；
6. 页面展示的版本与目标 SHA 一致。

预览通过后再发布生产。

### 6. 绑定域名

在平台先添加域名，再按平台给出的当前记录修改 DNS。不要从博客或另一个项目复制 CNAME/A 记录。等待平台显示域名验证和证书就绪，再切换正式流量。

### 7. 建立更新与回滚

日常流程：

```text
本地测试
→ GitHub 提交
→ 预览部署
→ 核心功能冒烟
→ 提升生产
→ 正式域名复测
→ 观察日志
```

记录如何把生产别名或流量切回上一成功部署。不要只知道“重新部署最新提交”。

## 海外 VPS 完整流程

海外 VPS 与国内自管服务器共享 Git、进程守护、Nginx/Caddy、DNS 和证书方法，但先补充：

1. 按主要用户选择区域和网络线路；
2. 检查公网 IPv4/IPv6、云防火墙和出站限制；
3. 测试 VPS 到 GitHub、包源、数据库和业务 API；
4. 使用普通部署用户和只读 Deploy Key；
5. 让应用监听回环地址，由 Nginx/Caddy 统一公开 80/443；
6. 配置 systemd、PM2 或容器重启策略；
7. 验证自动安全更新、日志轮转、磁盘、备份和监控；
8. 从目标国家/地区而不只是开发者本机做最终测试。

具体命令读取 [command-patterns.md](command-patterns.md)，但必须替换占位符并逐步验证。

## 区域、数据与网络

- 应用与数据库距离通常比应用与静态 CDN 距离更影响动态请求延迟。
- 跨区域写数据库会增加延迟和故障面；除非有明确架构，不要随意跨洲拆分。
- 上传文件优先放对象存储，不依赖平台临时磁盘。
- 为全球用户使用 CDN 时，仍需确认动态 API、认证和数据存储的实际区域。
- 面向中国大陆用户时，从大陆网络测试 DNS、TLS、静态资源和第三方脚本；不要只看海外测速。
- 选择区域前核验供应商当前可用区、数据驻留选项和迁移能力。

## 域名与 HTTPS

- 平台可能要求 A、AAAA、CNAME、ALIAS/ANAME 或 TXT 验证，必须采用当前项目页面提供的值。
- 根域和子域的记录类型可能不同。
- 不同时在多个平台声明同一生产域名，除非正在执行有计划的切换。
- 平台托管证书时观察证书状态，不重复运行 Certbot。
- VPS 使用 ACME 时，先证明 DNS 和验证入口可达，再申请证书并测试续期。
- 迁移域名时同时处理旧 HTTPS 证书、301 跳转、OAuth 回调、Cookie 域和邮件链接。

## 合规与官方核验

国外托管不等于没有合规要求。根据用户、主体、数据和业务核验：

- 隐私政策、服务条款、Cookie/追踪同意；
- 数据收集最小化、保留、删除和用户请求流程；
- 数据驻留、跨境传输和供应商数据处理条款；
- 支付、税务、消费者保护、版权和内容规则；
- 儿童、医疗、金融等特定场景要求。

不要凭技能内容给出法律结论。执行时优先查目标国家/地区主管机构、实际平台、域名注册商、支付商和数据处理供应商的当前官方资料。
