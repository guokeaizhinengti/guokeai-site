## 角色分工

把上线链路理解为一组职责，而不是一串产品名：

| 角色 | 负责什么 | 不负责什么 |
|---|---|---|
| 本地开发环境 | 开发、测试、产出提交 | 24 小时公网服务 |
| GitHub | 版本、协作、审计、发布源 | 自动成为生产运行环境 |
| 托管平台 | 构建、运行、平台入口、预览和回滚 | 替代版本事实源或业务数据设计 |
| 云服务器/VPS | 提供可控的主机运行环境 | 自动完成进程守护、代理和安全配置 |
| DNS | 把域名映射到入口 | 启动应用或开放端口 |
| 云安全组/防火墙 | 控制流量能否进入 | 路由页面和 API |
| Nginx/Caddy | TLS、静态文件、反向代理、跳转 | 业务处理 |
| 应用进程 | 业务逻辑和健康状态 | 公网域名解析 |
| 进程管理器 | 崩溃恢复、开机启动、日志入口 | 修复应用错误 |
| TLS 证书 | 证明域名身份并加密传输 | 证明业务合规 |

## 从本站成功经验抽象出的原则

本站采用 React/Vite 前端、Express 后端、GitHub、Ubuntu、Nginx、进程守护、DNS 和 Let's Encrypt。其价值不在这套具体技术，而在以下方法：

| 真实事件 | 抽象原则 | 通用做法 |
|---|---|---|
| 本地和服务器分别访问 GitHub | 身份按参与者隔离 | 本地用个人写权限，服务器用单仓库只读 Deploy Key |
| 域名曾有一处拼写错误 | 身份信息必须单一来源 | 在部署卡中只保留一个规范域名，配置前逐字符核对 |
| 服务器内部返回 200，公网仍超时 | 内部健康不等于外部可达 | 从进程、入口、系统防火墙、云安全组到公网逐层测试 |
| 前端使用 `/api`，Nginx 分流 | 同源入口降低环境耦合 | 浏览器只认正式域名，入口把 API 转到内部服务 |
| Nginx 同时服务静态站与 API | 边缘入口统一职责 | TLS、跳转、静态资源和反代集中在一个可验证入口 |
| 后端在代理后做限流 | 应用必须理解代理拓扑 | 正确传递转发头，并只信任明确的代理层 |
| 先打通 DNS/80 再申请证书 | 有依赖的步骤必须按序执行 | 先满足域名控制验证，再启用 HTTPS 和自动续期 |
| 旧域名跳转到新域名 | 变更需要兼容窗口 | 保留旧域名证书与 301 跳转，验证路径和查询串 |
| 线上持续出现平台下载差异 | 上线是反馈循环，不是终点 | 保留日志、健康检查、可回滚提交和快速发布路径 |

最关键的成功经验是“证据驱动的分层排障”：在当前层未被证实前，不越层猜测。

## 架构选择

按项目实际复杂度选择最小可持续架构：

- 纯静态站：静态托管/CDN、对象存储或 Nginx 静态目录即可。
- SPA + API：统一域名入口，静态目录加 `/api` 反代。
- SSR/单体应用：入口反代到受守护的应用进程，静态资源单独缓存。
- 多服务/高可用：容器和编排只有在发布、隔离或扩缩容收益明确时再引入。
- 有数据库/上传：先设计持久化、备份、迁移和恢复，再谈无状态发布。

不要为了“像生产”堆技术。优先减少故障面，同时保留日志、重启、回滚和自动续期。

## 完成定义

上线完成必须同时满足：

1. 可重复：从仓库指定 SHA 能重新部署。
2. 可访问：外部网络通过正式域名和 HTTPS 访问。
3. 可用：真实核心功能而非只有首页通过。
4. 可恢复：平台能重新部署目标版本，或服务器/进程重启后恢复。
5. 可诊断：日志和健康检查能定位故障层。
6. 可回滚：能恢复到上一可运行版本。
7. 可持续：证书、备份、监控、备案/许可与更新流程有人负责。
