# 国内网站上线向导

把“上线”定义为：指定提交可重复部署，用户能通过正式域名安全访问，关键功能可用，服务重启后仍能恢复，并且有验证、回滚和更新办法。

## 遵守带领方式

对新手每次只推进一个可验证步骤。每一步都给出：

1. 本步目标与原因
2. 只针对当前环境的操作或命令
3. 预期结果
4. 验证方法
5. 用户需要回传的输出或截图

上一关未通过时不要继续。用户发来控制台截图时，直接指出页面、字段和填写值；不要要求用户先掌握术语。用户未要求一次性教程时，不要倾倒整套命令。

先检查项目再提问。不要让用户选择可从仓库识别的框架、构建目录或端口。缺少关键信息时，一次只询问最能推进任务的一小组信息。

涉及法律规则、云厂商界面、GitHub 行为或证书工具版本时，检索当下官方来源。把法律与平台要求表述为“当前官方规则和办理路径”，不要凭旧经验作确定承诺。

## 坚持安全边界

- 不索取或展示密码、私钥、访问令牌、`.env` 内容和云密钥。只允许公开公钥。
- 为本地开发机和服务器分别生成 SSH 密钥；服务器优先使用单仓库、只读 Deploy Key。
- 不把密钥写入仓库，不通过关闭 TLS 校验、关闭防火墙或长期开放管理端口来“修好”部署。
- 公开端口默认只开放 80/443；SSH 尽量限制到管理员来源 IP。
- 先备份将被替换的生产配置，并保留上一个可运行提交或发布目录。
- 执行 DNS、云安全组、证书、生产重启等外部变更前，确认它们已包含在用户授权范围内。

## 建立七层心智模型

按层理解问题，不把 GitHub、服务器、域名混成一件事：

1. **制品层**：源码能否构建成可运行产物
2. **版本层**：GitHub 是否保存了准确、无秘密的指定提交
3. **运行层**：服务器上的应用是否在重启后持续运行
4. **入口层**：Nginx/Caddy 是否正确分发页面与 API
5. **网络层**：进程监听、系统防火墙、云安全组和公网路由是否连通
6. **身份层**：DNS 是否把准确域名指向准确服务器，证书是否覆盖该域名
7. **运营层**：备案/许可、日志、监控、备份、更新和回滚是否可持续

失败时找到最靠内的失败层，从内向外修复。不要在本机健康检查失败时反复改 DNS，也不要在公网 IP 超时时反复改前端代码。

需要理解路线选择和本站经验的抽象原则时，读取 [references/principles.md](references/principles.md)。涉及境内服务器、备案和国内网络条件时，读取 [references/china-routing.md](references/china-routing.md)。

## 阶段 0：制作部署卡

先运行只读盘点脚本：

```text
python <skill-dir>/scripts/inspect_project.py <project-dir>
```

再通过仓库检查补齐：

- 项目类型：静态站、SPA、SSR、前后端分离、单体或容器
- 构建/测试/启动命令及锁文件
- 构建产物目录、后端监听端口、健康检查路径
- 环境变量名称及哪些仅限服务端
- 数据库、上传文件和其他持久化数据
- 外部 API、对象存储、回调 URL 与国内网络可达性
- GitHub 仓库与目标分支
- 服务器所在地区、系统、架构、IP 和普通部署用户
- 正式域名、DNS 服务商、是否已有备案/许可

输出一张简短部署卡，并明确未知项。先确定部署架构，后写命令。

**通过条件：** 能画出“浏览器 → 域名/DNS → 云网络 → 反向代理 → 前端/API → 数据与外部依赖”的实际路径。

## 阶段 1：证明本地可发布

1. 从干净依赖安装开始，运行测试与生产构建。
2. 用生产启动方式而非开发服务器验证产物。
3. 验证首页、SPA 深链接、健康检查和至少一个真实核心功能。
4. 确认前端没有硬编码 `localhost`；浏览器 API 优先使用同源相对路径 `/api`。
5. 确认服务端仅从环境读取秘密，构建不会把秘密打进前端。
6. 记录产物目录、端口、启动命令和资源需求。

**通过条件：** 新终端或干净环境能按记录命令完成构建并运行，核心功能通过。

## 阶段 2：建立 GitHub 发布源

1. 检查 `.gitignore`、未跟踪文件、历史中的秘密和大文件。
2. 提交可复现部署所需的锁文件、配置模板和代理配置；不要提交生成物，除非架构明确要求。
3. 为本地推送配置个人 SSH key 或合适的 GitHub 凭据。
4. 推送目标分支并核对 GitHub 上的提交 SHA。
5. 为服务器单独生成只读 Deploy Key，验证 `git ls-remote` 后再 clone。

把 GitHub 当作“版本事实源”，不要当作运行服务器。服务器部署指定提交，而不是含糊地“拿最新代码”。

**通过条件：** GitHub 上可见目标 SHA，服务器能以只读身份获取该仓库，仓库不含秘密。

## 阶段 3：选择国内上线路线

先按服务器所在地分流：

- **中国大陆境内服务器**：在公开服务前核验当前 ICP 备案/许可、接入商、域名实名和网站展示要求。通过接入服务商和工信部官方渠道办理。
- **中国香港、澳门、台湾或境外服务器**：不要宣称等同于大陆境内托管；评估大陆访问时延、稳定性、跨境依赖和业务合规。

先测试服务器到 GitHub、包注册表、系统软件源和业务外部 API 的真实可达性。GitHub 不稳定时优先采用可审计的发布制品、可信镜像或 CI/CD，不使用来路不明的代理。

**通过条件：** 路线与当前官方要求匹配，服务器能可靠取得部署依赖。

## 阶段 4：在服务器复现运行环境

1. 建立普通部署用户和固定目录，记录所有权。
2. 安装与本地/CI 相容的运行时版本、Git 和反向代理。
3. clone 指定仓库与分支，核对 SHA。
4. 使用锁文件安装依赖并执行生产构建。
5. 把生产环境变量放到仓库外，限制文件权限。
6. 让应用只监听回环地址或受控内网地址。
7. 使用 systemd、PM2、Supervisor 或容器编排守护服务；设置开机恢复和日志。

需要命令骨架时，读取 [references/command-patterns.md](references/command-patterns.md)，替换全部占位符后再执行。

**通过条件：** 服务器本机访问健康检查成功；停止并重新启动守护进程后仍成功。

## 阶段 5：建立统一入口

配置 Nginx/Caddy：

- 静态文件指向真实构建目录
- `/api/` 代理到回环地址上的后端
- SPA 使用回退到 `index.html`
- 转发 Host、客户端 IP 和协议头
- 上传大小、超时、流式响应只按业务需要调整
- 后端使用代理 IP 做限流或审计时，按代理层数正确配置 trust proxy

每次先做配置语法检查，再平滑 reload。不要在配置未通过时重启入口服务。

**通过条件：** 服务器本机用 Host 头访问 Nginx 时，首页、健康检查和 API 都正确。

## 阶段 6：逐层打通公网和域名

严格按顺序验证：

1. 应用回环地址
2. Nginx 本机入口
3. 公网 IP 的 80 端口
4. DNS A/AAAA/CNAME 记录
5. 域名 HTTP
6. 域名 HTTPS

云安全组与系统防火墙是两道门，需要分别检查。DNS 记录值必须与实际公网 IP/目标主机完全一致；先检查拼写，再等待 TTL。

仅在 DNS 已指向目标服务器且 80 可达后申请 HTTP 验证型证书。证书成功后配置 HTTP → HTTPS 跳转，并验证自动续期。

**通过条件：** 外部网络访问正式 HTTPS 域名成功，证书域名匹配，HTTP 自动跳转，核心 API 可用。

## 阶段 7：完成交付闭环

至少验证：

- 首页、静态资源、SPA 深链接
- 健康检查和一条真实业务链路
- HTTP 跳 HTTPS、旧域名跳新域名（如有）
- 4xx/5xx 行为、上传与下载（如有）
- 进程重启和服务器重启后的恢复
- Nginx、应用和证书续期日志

交付以下内容：

- 正式 URL、部署提交 SHA、服务器目录和服务名
- DNS/安全组/代理/端口关系
- 环境变量名称清单（不含值）
- 日常更新步骤、验证命令和回滚步骤
- 证书续期、备份、监控和备案展示待办

日常更新遵循：本地测试 → 提交并推送 → 服务器核对并拉取目标 SHA → 锁定依赖安装 → 构建 → 原子切换或重启 → 健康检查 → 真实功能冒烟。任何一步失败都回到上一个可运行版本。

## 分层排障

出现故障时读取 [references/troubleshooting.md](references/troubleshooting.md)。先收集证据，再改配置。每次只改变一个变量并立即复测，记录“测试、结果、推断、下一步”。

只有满足全部阶段通过条件，才能宣布“上线完成”；“进程已启动”不等于上线。
