# 国内外网站上线向导

把“上线”定义为：指定提交可重复部署，目标用户能通过正式域名安全访问，关键功能可用，服务中断后能恢复，并且有验证、更新和回滚办法。

## 第一句先选择路线

如果用户没有明确说国内或国外，第一条回复只问：

```text
你希望走哪条上线线路？

1. 国内上线：中国大陆境内云服务器，包含备案、云安全组、Nginx 和 HTTPS。
2. 国外上线：海外托管平台或海外服务器，包含平台选择、全球区域、域名和 HTTPS。

回复“国内”或“国外”即可。
```

不要在用户选择前检查项目、推荐平台或给出部署命令。用户已明确“中国大陆服务器、国内云、海外平台、Vercel、Cloudflare、海外 VPS”等目标时，直接判定路线，不重复提问：

- 中国大陆境内服务器归入**国内路线**。
- 中国香港、澳门、台湾及其他境外地区的托管平台或服务器归入**国外路线**。
- 主要用户在中国大陆但服务器在境外时，仍走国外路线，同时评估跨境访问质量。

选定路线后先复述：“已选择国内/国外上线，下一步先盘点项目。”此后不要在未得到用户同意时擅自切换路线。

## 遵守带领方式

对新手每次只推进一个可验证步骤。每一步都给出：

1. 本步目标与原因
2. 只针对当前环境的操作或命令
3. 预期结果
4. 验证方法
5. 用户需要回传的输出或截图

上一关未通过时不要继续。用户发来控制台截图时，直接指出页面、字段和填写值；不要要求用户先掌握术语。用户未要求一次性教程时，不要倾倒整套命令。

路线确认后，先检查项目再追问技术信息。不要让用户选择可从仓库识别的框架、构建目录或端口。缺少关键信息时，一次只询问最能推进任务的一小组信息。

涉及法律规则、云厂商界面、托管平台行为、GitHub 行为或证书工具版本时，检索当下官方来源。把法律与平台要求表述为“当前官方规则和办理路径”，不要凭旧经验作确定承诺。

## 坚持安全边界

- 不索取或展示密码、私钥、访问令牌、`.env` 内容和云密钥。只允许公开公钥。
- 本地开发机、托管平台和服务器使用各自独立的最小权限身份。
- 自管服务器优先使用单仓库、只读 Deploy Key；托管平台优先使用官方 GitHub App/OAuth 集成。
- 不把密钥写入仓库，不通过关闭 TLS 校验、防火墙或长期开放管理端口来“修好”部署。
- 自管服务器公开端口默认只开放 80/443；SSH 尽量限制到管理员来源 IP。
- 先备份将被替换的生产配置，并保留上一个可运行提交或发布版本。
- 执行 DNS、云安全组、平台生产发布、证书或生产重启等外部变更前，确认它们已包含在用户授权范围内。

## 建立七层心智模型

按层理解问题，不把 GitHub、托管平台、服务器和域名混成一件事：

1. **制品层**：源码能否构建成可运行产物
2. **版本层**：GitHub 是否保存准确、无秘密的指定提交
3. **运行层**：平台或服务器是否持续运行该版本
4. **入口层**：平台入口或 Nginx/Caddy 是否正确分发页面与 API
5. **网络层**：平台网络、进程监听、防火墙和公网路由是否连通
6. **身份层**：DNS 是否指向准确入口，证书是否覆盖该域名
7. **运营层**：合规、日志、监控、备份、更新和回滚是否可持续

失败时找到最靠内的失败层，从内向外修复。不要在构建失败时反复改 DNS，也不要在平台环境变量缺失时反复修改域名。

需要理解通用原则时读取 [references/principles.md](references/principles.md)。选择国内路线时读取 [references/china-routing.md](references/china-routing.md)；选择国外路线时读取 [references/global-routing.md](references/global-routing.md)。只加载当前路线需要的参考资料。

## 阶段 0：制作部署卡

先运行只读盘点脚本：

```text
python <skill-dir>/scripts/inspect_project.py <project-dir>
```

再通过仓库检查补齐：

- 已选择的路线：国内或国外
- 项目类型：静态站、SPA、SSR、前后端分离、单体或容器
- 构建/测试/启动命令及锁文件
- 构建产物目录、后端监听端口、健康检查路径
- 环境变量名称及哪些仅限服务端
- 数据库、上传文件、后台任务和其他持久化数据
- 外部 API、对象存储、回调 URL 与目标区域可达性
- GitHub 仓库与目标分支
- 目标用户、计划区域、托管方式和预算边界
- 正式域名、DNS 服务商与当前合规状态

国内路线补充服务器系统、架构、公网 IP、普通部署用户和备案状态。国外路线补充“托管平台或自管 VPS”、期望区域、平台账号和是否需要常驻进程/WebSocket/定时任务。

输出一张简短部署卡并明确未知项。先确定部署架构，后写命令。

**通过条件：** 能画出“浏览器 → 域名/DNS → 平台或云网络 → Web 入口 → 前端/API → 数据与外部依赖”的实际路径。

## 阶段 1：证明本地可发布

1. 从干净依赖安装开始，运行测试与生产构建。
2. 用生产启动方式而非开发服务器验证产物。
3. 验证首页、SPA 深链接、健康检查和至少一个真实核心功能。
4. 确认前端没有硬编码 `localhost`；浏览器 API 优先使用同源相对路径 `/api`。
5. 确认服务端仅从环境读取秘密，构建不会把秘密打进前端。
6. 记录产物目录、端口、启动命令、运行时版本和资源需求。

**通过条件：** 新终端或干净环境能按记录命令完成构建并运行，核心功能通过。

## 阶段 2：建立 GitHub 发布源

1. 检查 `.gitignore`、未跟踪文件、历史中的秘密和大文件。
2. 提交可复现部署所需的锁文件、配置模板和部署配置；不要提交秘密。
3. 为本地推送配置个人 SSH Key 或合适的 GitHub 凭据。
4. 推送目标分支并核对 GitHub 上的提交 SHA。
5. 托管平台使用官方 GitHub 集成并仅授权目标仓库；自管服务器使用独立的只读 Deploy Key。

把 GitHub 当作“版本事实源”，不要当作运行服务器。平台或服务器部署指定提交，而不是含糊地“拿最新代码”。

**通过条件：** GitHub 上可见目标 SHA，部署端能以最小权限取得该仓库，仓库不含秘密。

## 阶段 3：进入对应上线分支

### 国内路线

按 [references/china-routing.md](references/china-routing.md) 确认：

1. 服务器是否在中国大陆境内；
2. 当前 ICP 备案/许可、接入商、域名实名和展示要求；
3. 服务器到 GitHub、包注册表和外部 API 的真实可达性；
4. 云安全组、系统防火墙和后续 HTTPS 入口。

不要等技术部署结束后才检查备案。GitHub 不稳定时使用可审计的发布制品、可信镜像或 CI/CD，不使用来路不明的代理。

### 国外路线

按 [references/global-routing.md](references/global-routing.md) 先选托管方式：

- **托管平台**：适合静态站、SPA、常见 SSR、Serverless 和支持该运行时的 Web 服务。优先用于第一次海外上线。
- **自管 VPS**：适合必须常驻的进程、特殊系统依赖、复杂网络、完整主机控制或平台限制无法满足的项目。

再按目标用户选择区域，检查数据库、对象存储和第三方 API 是否处于相近区域。用户点名平台时只查该平台当前官方文档，不凭印象猜构建目录、价格、限制或 DNS 记录。

**通过条件：** 路线、托管方式、目标区域、运行时和数据依赖相容，并明确生产入口。

## 阶段 4：部署应用

### 托管平台

1. 通过官方 GitHub 集成导入目标仓库和分支。
2. 核对项目根目录、运行时版本、安装命令、构建命令和产物目录。
3. 在平台密钥界面添加生产环境变量，不把值写入仓库或构建日志。
4. 为数据库、对象存储和后台任务选择与应用相容的区域和持久化方案。
5. 先创建预览部署，核对部署日志、提交 SHA 和平台临时域名。
6. 验证核心功能后再提升为生产部署。

不要把本地开发服务器当成生产启动命令。不要假设所有平台都支持本地磁盘持久化、常驻进程、WebSocket 或任意执行时长。

**通过条件：** 平台临时域名能访问目标 SHA，重新部署同一提交结果一致，核心功能通过。

### 国内或国外自管服务器

1. 建立普通部署用户和固定目录，记录所有权。
2. 安装与本地/CI 相容的运行时版本、Git 和反向代理。
3. clone 指定仓库与分支，核对 SHA。
4. 使用锁文件安装依赖并执行生产构建。
5. 把生产环境变量放到仓库外，限制文件权限。
6. 让应用只监听回环地址或受控内网地址。
7. 使用 systemd、PM2、Supervisor 或容器编排守护服务；设置开机恢复和日志。

需要命令骨架时读取 [references/command-patterns.md](references/command-patterns.md)，替换全部占位符后再执行。

**通过条件：** 服务器本机访问健康检查成功；停止并重新启动守护进程后仍成功。

## 阶段 5：建立统一入口

### 托管平台

使用平台提供的入口和路由规则：

- 确认 SPA 回退、SSR 路由、API/函数路径和静态资源目录
- 检查预览环境与生产环境的变量差异
- 配置必要的重写、重定向、缓存和安全响应头
- 只在确有需要时接入 CDN、WAF 或自定义代理

不要在平台已经托管入口时额外配置一台 Nginx 服务器。

### 自管服务器

配置 Nginx/Caddy：

- 静态文件指向真实构建目录
- `/api/` 代理到回环地址上的后端
- SPA 使用回退到 `index.html`
- 转发 Host、客户端 IP 和协议头
- 上传大小、超时、流式响应只按业务需要调整
- 后端按明确代理层数配置 trust proxy

每次先做配置语法检查，再平滑 reload。

**通过条件：** 平台临时域名或服务器本机 Host 入口的首页、健康检查和 API 都正确。

## 阶段 6：绑定公网域名与 HTTPS

严格按当前托管方式验证：

### 托管平台

1. 在平台添加准确的自定义域名。
2. 使用平台当前给出的 A、AAAA、CNAME 或验证记录，不复制其他项目的值。
3. 等待平台确认 DNS 所有权与证书状态。
4. 验证主域、`www`/子域和跳转策略。
5. 平台自动管理证书时，不再手工申请重复证书。

### 自管服务器

1. 应用回环地址
2. Nginx/Caddy 本机入口
3. 公网 IP 的 80 端口
4. DNS A/AAAA/CNAME 记录
5. 域名 HTTP
6. 域名 HTTPS

云安全组与系统防火墙是两道门，需要分别检查。仅在 DNS 已指向目标服务器且验证入口可达后申请证书；成功后配置 HTTP → HTTPS 跳转并验证自动续期。

**通过条件：** 外部网络访问正式 HTTPS 域名成功，证书域名匹配，跳转策略正确，核心 API 可用。

## 阶段 7：完成交付闭环

至少验证：

- 首页、静态资源、SPA/SSR 深链接
- 健康检查和一条真实业务链路
- 正式域名、HTTPS 和旧域名跳转（如有）
- 4xx/5xx、上传、下载、后台任务和 WebSocket（如有）
- 平台重新部署，或自管服务器进程/系统重启后的恢复
- 平台构建日志、应用日志、入口日志和证书状态
- 目标地区真实网络的访问质量

交付以下内容：

- 正式 URL、部署提交 SHA、目标区域和托管方式
- 平台项目/服务名，或服务器目录、服务名和端口关系
- DNS 记录、环境变量名称清单（不含值）
- 数据库、对象存储、备份和恢复责任
- 日常更新、验证和回滚步骤
- 国内备案展示或国外隐私/条款等仍需核验的合规待办

托管平台更新遵循：本地测试 → 提交并推送 → 预览部署 → 冒烟测试 → 提升生产 → 正式域名复测 → 必要时回滚上一部署。

自管服务器更新遵循：本地测试 → 提交并推送 → 服务器拉取目标 SHA → 锁定依赖安装 → 构建 → 原子切换或重启 → 健康检查 → 真实功能冒烟。

任何一步失败都回到上一个可运行版本。

## 分层排障

出现故障时读取 [references/troubleshooting.md](references/troubleshooting.md)。先收集证据，再改配置。每次只改变一个变量并立即复测，记录“测试、结果、推断、下一步”。

只有满足当前路线的全部通过条件，才能宣布“上线完成”；“构建成功”或“进程已启动”都不等于上线。
